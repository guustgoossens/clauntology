# Ontolo GG — The Living Personal Ontology

## Vision

A living, self-updating personal knowledge graph that indexes every conversation you've ever had with Claude (web + Code), extracts a deep understanding of who you are — skills, interests, thinking patterns, beliefs, projects, relationships, psychological tendencies, growth trajectory — and presents it as an interactive knowledge environment.

It serves three purposes:
1. **Mirror** — See yourself mapped: what you know, how you think, where you're growing
2. **Tool** — Context you can plug into anything: therapy, coaching, hiring, self-improvement
3. **Compass** — A skill tree showing where you are, what's adjacent, what you could become

---

## The Homo Universalis Skill Map

### Philosophy

The Renaissance ideal of the "universal person" — someone who develops all human faculties. Not a job classification, but a complete map of everything a human can be capable of. Your conversations get mapped onto this, showing where you're deep, where you're developing, and where the white space is.

### Taxonomy Design (8 Domains, ~200 Skills)

The taxonomy is a tree: 8 root domains → ~40 skill families → ~200 individual skills. Each skill has a depth scale (0-5) derived from conversation evidence.

```
Depth 0: No evidence
Depth 1: Awareness — mentioned or asked about
Depth 2: Exploration — actively learning, asking questions
Depth 3: Application — using in projects, making decisions with it
Depth 4: Fluency — teaching others, deep understanding, creative application
Depth 5: Mastery — pushing boundaries, novel contributions, integrated into identity
```

### How Skills Get Scored

Each skill gets a depth score (0-5) computed from conversation evidence:

```python
score = weighted_sum(
    frequency,          # How often discussed (log-scaled)
    recency,            # When last active (exponential decay)
    depth_of_discourse, # Surface mention vs deep application
    teaching_signal,    # Did you explain it to Claude or ask Claude to explain?
    application_signal, # Did you use it in a project?
    evolution_signal,   # Is it growing over time?
)
```

Evidence is always linked: every score traces back to specific conversations and quotes.

## Data Architecture

### Source Data

| Source | Format | Volume | Content |
|--------|--------|--------|---------|
| Claude Web | `conversations.json` (64.6 MB) | 858 conversations, 6,669 messages | Clean chat: human/assistant text, thinking, tool use, voice notes, file attachments |
| Claude Web | `projects.json` (678 KB) | 18 projects | Project names, descriptions, system prompts |
| Claude Web | `memories.json` (22 KB) | Global + per-project memories | Structured memory about user |
| Claude Code | `~/.claude/projects/*/*.jsonl` (902 MB) | 33 projects, 1,749 sessions | Messages with tool calls, file reads, diffs, system prompts — needs filtering |

### Normalized Conversation Format

Both sources get normalized to a common format before extraction:

```typescript
interface NormalizedConversation {
  id: string;                    // UUID
  title: string;                 // Conversation name or generated title
  source: "claude_web" | "claude_code";
  platform_project: string;      // Web project name or Code project path
  created_at: string;            // ISO 8601
  updated_at: string;            // ISO 8601
  messages: NormalizedMessage[];
  metadata: {
    message_count: number;
    has_files: boolean;
    file_names: string[];
    has_voice: boolean;
    has_thinking: boolean;
    has_tool_use: boolean;
  };
}

interface NormalizedMessage {
  id: string;
  role: "user" | "assistant";
  text: string;                  // Extracted clean text content
  timestamp: string;
  attachments: string[];         // File names mentioned
}
```

### Processing Manifest

Tracks what's been processed for incremental updates:

```typescript
interface ProcessingManifest {
  last_updated: string;
  web_exports: {
    [zip_filename: string]: {
      hash: string;              // SHA-256 of zip file
      processed_at: string;
      conversation_count: number;
      conversation_ids: string[];
    };
  };
  code_sessions: {
    [project_path: string]: {
      [session_file: string]: {
        hash: string;
        processed_at: string;
        message_count: number;
      };
    };
  };
  extraction_cache: {
    [conversation_id: string]: {
      extracted_at: string;
      model_used: string;
      extraction_version: number;  // Bump when prompt changes
    };
  };
}
```

### Incremental Update Flow

```
New web export dropped in ontolo_gg/:
  1. Hash the zip file
  2. Compare against manifest.web_exports
  3. If new: extract, find conversations not in manifest
  4. Process only new conversations through extraction
  5. Upsert into graph
  6. Update manifest

New Claude Code sessions (daily hook or manual trigger):
  1. Scan ~/.claude/projects/ for .jsonl files
  2. Compare file paths + hashes against manifest.code_sessions
  3. Process only new/changed sessions
  4. Upsert into graph
  5. Update manifest
```

### The SkillNode Table — The Full Homo Universalis Map

The `SkillNode` table contains EVERY skill in the taxonomy (~200+), whether or not you have evidence for it. It's the complete map. The `Skill` table contains only skills with evidence from your conversations. They're linked via `MAPS_TO`.

This means:
- The full taxonomy is always visible (white space included)
- Your actual skills overlay onto it
- You can see gaps, adjacencies, and growth opportunities
- New skills emerge from conversations and get mapped to taxonomy nodes

## Extraction Pipeline

### Overview

Each conversation gets processed by Claude Opus to extract structured entities. This is the most expensive and most important part of the pipeline.

### Extraction Prompt Design

For each conversation, we send a prompt that:
1. Provides the full conversation text (user + assistant messages)
2. Asks for structured JSON output
3. Extracts multiple entity types simultaneously
4. Requires evidence (quotes) for every claim

```python
EXTRACTION_SYSTEM_PROMPT = """You are an expert psychologist, skill assessor, and knowledge analyst.
You are analyzing conversations between a person (Guust) and Claude AI to build a deep
understanding of who this person is.

For each conversation, extract the following as structured JSON:

1. TOPICS: What subjects were discussed? How deep did it go?
2. SKILLS: What skills were demonstrated or practiced? At what level?
3. PROJECTS: What projects were mentioned or worked on?
4. PEOPLE: Who was mentioned? In what context?
5. BELIEFS: What opinions, values, or worldview elements were expressed?
6. QUESTIONS: What did the person ask about? What are they curious about?
7. THINKING_PATTERNS: How does this person think? What cognitive patterns are visible?
8. EMOTIONAL_TONE: What was the emotional character of the conversation?
9. GROWTH_SIGNALS: What evidence of learning, development, or change?

For EVERY extraction, provide:
- A specific quote or reference from the conversation as evidence
- A confidence score (0.0-1.0) for your assessment
```

### Batch Processing Strategy

```
Total conversations: ~2,600 (858 web + ~1,749 Code sessions)
Estimated tokens per conversation: ~4,000 input + ~2,000 output (average)
Total estimated tokens: ~15.6M input + ~5.2M output

Strategy:
1. Process in batches of 20 conversations
2. Between batches, save progress to manifest
3. If interrupted, resume from last completed batch
4. Rate limit to avoid API throttling
5. Cache all extractions as JSON alongside manifest

Processing order:
1. Claude Web conversations (cleaner, richer personal content)
2. Claude Code sessions (technical, needs more filtering)
```

### Post-Extraction: Entity Resolution

After extraction, we need to merge duplicate entities across conversations:

```
"React" + "React.js" + "ReactJS" → single Topic node
"Accaio" + "the startup" + "AI accounting platform" → single Project node
"my girlfriend" + "her (architecture student)" → single Person node
```

This uses fuzzy string matching + LLM-assisted disambiguation for ambiguous cases.

## Automation & Sync

### Claude Code Hook (Post-Session)

After each Claude Code session, automatically process the new conversation:

```json
// ~/.claude/settings.json hook
{
  "hooks": {
    "PostSession": [{
      "command": "bun /Users/guustgoossens/Desktop/PerFin/Persoon/GG.ai/ontolo_gg/scripts/sync-code-session.ts",
      "description": "Sync new Claude Code session to Ontolo GG"
    }]
  }
}
```

The sync script:
1. Finds the just-completed session file
2. Checks manifest — if already processed, skip
3. Normalizes the conversation
4. Runs extraction (using Claude API)
5. Upserts into KuzuDB
6. Regenerates affected markdown pages
7. Updates manifest

### Web Export Import Script

Manual trigger when you drop a new export:

```bash
bun run ontolo_gg/scripts/import-web-export.ts
```

1. Scans `ontolo_gg/` for new zip files
2. Extracts and compares against manifest
3. Processes only new conversations
4. Full pipeline: normalize → extract → graph → markdown

### Daily Aggregation (Optional Cron)

A lighter daily job that:
1. Recomputes skill scores (may change with time decay)
2. Updates trajectory labels (growing/stable/declining)
3. Regenerates dashboard stats
4. Detects new patterns across recent conversations

## Key Design Decisions

### Why KuzuDB (Embedded) Over Neo4j/Cloud
- No server to run — just a local directory
- Fast for this data size (~10K nodes, ~100K edges max)
- Cypher compatibility — same query language
- WASM option for future browser-based version
- You already know it and love it

### Why Markdown Vault as Source of Truth
- Human-readable without any tooling
- Version-controllable with git
- Openable in Obsidian immediately
- Portable — not locked into any system
- The web UI reads from it (or from KuzuDB for graph queries)

### Why Claude Opus for Extraction
- Highest quality extraction — this is the core value
- You're paying for it already
- Running overnight via loops amortizes cost
- Quality >> speed for this use case

### Why Hybrid Skill Taxonomy
- Standard taxonomies are too shallow and job-focused
- Pure emergent is too messy and hard to visualize
- Hybrid gives you the complete map (aspirational) + deep detail (actual)
- You can see yourself in context of all human capability
